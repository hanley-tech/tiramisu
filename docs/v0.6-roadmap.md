# v0.6+ Roadmap ‚Äî the AI release wave

Written 2026-05-10, after a model-landscape research pass. The image-AI
field has moved a lot in the six months between v0.4 and v0.5; this doc
captures what's available, what fits Tiramisu's PXLayer model, and what
order to ship them in.

**See also: [/docs/creator-workflows-2026.md](./creator-workflows-2026.md)**
‚Äî the strategic doc on what an editor *becomes* when AI is a primary
input rather than a finishing-pass panel. The "Reimagine" loop described
there is now the third v0.6 thread alongside Auto-Layer and Smart Select 2.

The previous v0.6 plan (SAM 2 + magnetic lasso + soft selections) is
**superseded by this doc**. SAM 2 is replaced by SAM 3; magnetic lasso
is demoted to v0.7 (mostly obsoleted by what AI selection can now do).

---

## Headline: v0.6 ‚Äî Auto-Layer + Smart Select 2 + Reimagine v0

Two AI epics. Both ship as opt-in Core ML downloads alongside the
existing Vision-based features (Vision stays default ‚Äî instant, no
download, neural-engine).

### 1. Auto-Layer ‚Äî Qwen-Image-Layered (Apache 2.0, Dec 2025)

**The pitch:** drop in a flat photo ‚Üí out comes a multi-layer Tiramisu
document. Each layer is a real RGBA `PXLayer`, independently editable,
with **plausible reconstructed pixels behind occluded regions** (the
background isn't person-shaped ‚Äî it's a real background where the
person used to be). Photoshop doesn't ship this. Affinity doesn't.
Pixelmator doesn't. We'd be the first mainstream Mac editor with it.

**Why it fits:** our `PXLayer` model already handles arbitrary RGBA
raster layers. The output of Qwen-Image-Layered is N RGBA images. The
integration is mostly: run model ‚Üí write each output as a `PXLayer(kind: .raster)` ‚Üí stack them in z-order ‚Üí make the topmost active.

**Architecture (from the paper):**

- RGBA-VAE ‚Äî unifies latent representations of RGB and RGBA.
- VLD-MMDiT ‚Äî Variable-Layer Decomposition Multi-Modal DiT. Decides
  how many layers to produce per image.
- Multi-stage training pipeline that mined real Photoshop PSDs to learn
  what "good" decomposition looks like.

**Engineering shape:**

1. Convert Qwen-Image-Layered to Core ML (likely 1‚Äì2 GB packaged).
2. Ship as opt-in download via the same mechanism we use for the Local
   FLUX-Fill backend (`mflux-generate-fill`-style subprocess, or a
   bundled `.mlpackage` if size allows).
3. New menu item: **AI ‚Üí Auto-Layer Image** (default `‚åò‚áßL`).
4. New `AutoLayerService.swift` parallel to `LocalFluxFillService`,
   with a single `decompose(image:) async throws -> [CGImage]`.
5. UI flow: progress sheet ("Decomposing into layers‚Ä¶ N seconds") ‚Üí
   layers panel populates with N new RGBA layers, each named for its
   semantic content if Foundation Models is wired up (see v0.7).
6. Recursive decomposition supported by the model ‚Äî UI affordance:
   right-click a layer ‚Üí "Auto-Layer this layer further".

**Risk:** model size on disk (~1‚Äì2 GB), conversion friction (Qwen-Image
already runs on PyTorch + Diffusers; need to validate the Core ML
conversion path or fall back to a `coremltools.convert` flow).

### 2. Smart Select 2 ‚Äî SAM 3 / SAM 3.1 (Nov 2025 / Mar 2026)

Replaces the v0.6 plan for SAM 2 *entirely*. SAM 3 is strictly better:

- **Text-prompt selection** ("select all chairs in this photo") via
  Promptable Concept Segmentation. New for the SAM family.
- **2√ó perf gain** vs SAM 2 in PCS, while keeping interactive
  click-prompt segmentation.
- **SAM 3.1** (March 2026) added object multiplexing for video.

For our (image-only) workflow, SAM 3 fits in the same shape as the
planned SAM 2 work:

- Convert SAM 3 image-encoder + mask-decoder to Core ML.
- Ship as opt-in download (~150‚Äì300 MB depending on size variant).
- Replace the existing `Tool.smartSelect` handler in TransformOverlay
  to call SAM 3 first; fall back to Vision foreground-instance if
  SAM 3 isn't installed.
- New tool option: **Text prompt** field. Accepts free text; produces
  one or more concept-level selection paths.
- Click prompts and box prompts also supported (SAM's original API).

**Risk:** the SAM 3 reference impl uses HuggingFace transformers. Core
ML conversion is feasible (community has done it for SAM 2) but has to
be re-done for the new architecture (presence token, decoupled
detector‚Äìtracker). Allocate a day for conversion validation.

### 3. Reimagine v0 ‚Äî the creator workflow headline

**Why this is in v0.6, not v0.7:** see [creator-workflows-2026.md](./creator-workflows-2026.md)
for the full thesis. Short version: the creator's actual workflow today
is "rough draft ‚Üí AI reimagine ‚Üí ship." Tools that don't support this
loop natively force creators to round-trip into Krea / Magnific /
Midjourney and lose layer structure. Tiramisu can be the first editor
where Reimagine is a peer to the brush tool.

**v0 scope (intentionally tight):**

- **Reimagine Canvas** (`‚åò‚áßR`) ‚Äî img2img against the existing
  generative-fill backend (Replicate or LocalFlux). Uses the entire
  composite as the reference. Strength slider (0‚Ä¶1). Prompt field.
  Result lands as a new raster layer above the current stack.
- **Reimagine Selection** ‚Äî same as above but scoped to
  `selectionPath`. Replaces only the selected region; pixels outside
  are byte-identical.
- **Lineage panel (basic)** ‚Äî one new layer per Reimagine attempt,
  named with the prompt + a generation index. No tree UI yet ‚Äî just
  ordered layers in the panel. Tree visualization is a v0.7 polish.

**v0 is intentionally NOT:**

- Subject lock / style reference / depth lock (Tier 2 in the creator
  doc) ‚Äî needs IP-Adapter / ControlNet wiring, ~3-5 days. Defer to
  v0.6.1 or v0.7.
- Variations panel ‚Äî defer to v0.7.
- Recipes / templates ‚Äî defer to v0.7.

**Engineering shape:**

- Extend `GenerativeFillService` with an `Intent` case for Reimagine
  (different from `.fill` / `.expand` / `.remove`). The mask is the
  whole canvas (or the selection), the input is the composite, the
  prompt comes from the user.
- Reuse the existing `LocalFluxFillService` and `ReplicateFillService`
  switch ‚Äî both backends already support img2img with strength.
- New menu item: **AI ‚Üí Reimagine Canvas / Selection** under the
  existing AI menu.
- New side panel or sheet: prompt + strength + (later) reference image.

~2-3 days for v0.

### 4. (Optional, lighter) Soft / feathered selections

Required for v0.6 only if the new AI selections feel wrong without
feathered edges (likely they will ‚Äî both Auto-Layer, SAM 3, and
Reimagine Selection produce soft-edged masks).

- Add `selectionMask: CGImage?` alongside `selectionPath` on
  DocumentStore.
- PaintStroke clip uses the mask if present, the path otherwise.
- Refine Edge gains a true Feather slider that operates on the mask.

~1 day if scoped tight.

## v0.7 menu ‚Äî drop-in upgrades + new features

These are smaller, mostly-pluggable improvements. Pick a subset for v0.7
based on what feels missing after v0.6.

### Drop-in model upgrades (no UI changes)

| Current                         | Upgrade                                              | Why                                                                                      |
|---------------------------------|------------------------------------------------------|------------------------------------------------------------------------------------------|
| Vision foreground (BG removal)  | RMBG 2.0 (BiRefNet-derived, Bria, ~80 MB)            | Beats Vision + Photoshop on complex backgrounds. Opt-in alongside Vision.                |
| FLUX.1 Fill (LocalFlux backend) | FLUX.2 (Black Forest Labs, Nov 2025)                 | Better prompt adherence, text rendering, anatomy. Same workflow.                         |
| DepthAnything v2 (StudioRelight) | Depth Anything 3 (ICLR 2026)                        | Single-transformer, beats DA2 in monocular depth. Cleaner relight around hair / edges.  |

### New features

- **Magic Eraser** ‚Äî SAM 3 produces a mask, LaMa fills the hole. Click
  an object ‚Üí it disappears with plausible reconstructed background.
  Photoshop calls this "Spot Healing"; modern phones call it "Magic
  Eraser." LaMa runs locally on Apple Silicon, no diffusion required.
- **AI Upscale** ‚Äî Real-ESRGAN converted to Core ML. ~78√ó speedup
  vs PyTorch CPU on Apple Silicon. New panel item or menu command:
  upscale a layer 2√ó / 4√ó in place. SUPIR is higher-quality but slow
  (diffusion-based) ‚Äî good v1.0 target, not v0.7.
- **Restore Faces** ‚Äî CodeFormer (heavy degradation) + GFPGAN
  (identity preservation) combined workflow. Useful especially with
  our existing Skin Retouch ‚Äî a complete "old photo restoration"
  pipeline.
- **Prompt helper** (Apple Foundation Models framework, on-device
  3B-param LLM in macOS 26) ‚Äî improve user prompts in Generative Fill,
  auto-name generated/auto-layered layers, suggest "this image looks
  decomposable, want to Auto-Layer it?". Free, on-device, no download.

### Deferred / probably not worth it

- **Magnetic Lasso** (Mortensen & Barrett 1995 live-wire) ‚Äî was on the
  v0.6 list; demoted because Smart Select 2 with click + text prompts
  covers ~all the cases magnetic lasso was for. Keep as a "Photoshop has
  it" checkbox if a user explicitly asks.
- **SAM 3D** ‚Äî Meta also shipped a 3D variant. Not relevant for a 2D
  editor. Skip.
- **Text2Layer / MuLAn / LayerDecomp** ‚Äî earlier-generation
  layer-decomposition research papers. Qwen-Image-Layered is the
  current SOTA so they're moot.
- **LayerDiffuse** ‚Äî generates *new* transparent layers from text
  prompts. Different problem from decomposition; could be useful for a
  future "AI sticker / asset generator" feature but doesn't supersede
  Auto-Layer.

## Strategic notes

**Why Auto-Layer is the right v0.6 headline.** Our brand is "free,
AI-native alternative to Photoshop." Every release so far has been
either a parity-with-Photoshop feature (paint, selection) or a small
AI-on-the-margin feature (BG removal, gen fill, skin retouch). Auto-
Layer is the first feature where we'd be **categorically ahead** of
Adobe ‚Äî they don't ship anything like it. That's the marketing beat.

**Why ship SAM 3 alongside Auto-Layer.** Vision-based Smart Select
shipped in v0.5 only handles instance-level segmentation; users who
discover this limitation will look elsewhere. SAM 3 answers it
completely. Pairing with Auto-Layer makes v0.6 cohesive ("the AI
release") rather than just "more selection tools."

**Why opt-in download.** Both models add 1‚Äì2 GB+ to disk. Bundling
forces a 2 GB DMG that scares off casual users. Same pattern as
LocalFlux: ship a small "AI extras" download manager, let users opt
into the model that fits their workflow.

**The v0.7 menu is intentionally a menu.** After v0.6 ships we'll have
real telemetry on what users want. Pick from the list based on what's
missing, don't pre-commit.
