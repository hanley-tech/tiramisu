# Tiramisu — agent onboarding

Native macOS thumbnail / image editor. Free, open source, AI-native.

> Strategy, vision, design direction, architecture write-ups, and the marketing
> site live in a sibling repo: **[hanley-tech/tiramisu_www](https://github.com/hanley-tech/tiramisu_www)**
> (also published at **tiramisu.hanley.world**). Read those for the *why*. This
> repo is the *how* — the Swift code only.

## Hard rules (don't violate)

- **Bundle ID is `world.hanley.tiramisu`**. Don't reintroduce `ai.taiso.*` anywhere.
- **Project is generated by xcodegen** from `project.yml`. Never edit `Tiramisu.xcodeproj/` by hand — run `xcodegen generate` instead.
- **Sandbox is intentionally OFF** — the app spawns the user-installed `mflux-generate-fill` subprocess for the Local FLUX-Fill backend, which sandbox would block. Don't add `com.apple.security.app-sandbox` to entitlements.
- **No fallback logic that masks errors** — surface real errors with logging.
- **Active scheme matters.** `project.yml` defines a single target: `Tiramisu` (the app). When you Run from Xcode, make sure the scheme is `Tiramisu`.

## Build / run

```bash
xcodegen generate                                              # after editing project.yml
xcodebuild -project Tiramisu.xcodeproj -scheme Tiramisu build  # build app
```

In Debug, a post-build phase installs to `~/Applications/`. The app launches a debug `ControlServer` on `127.0.0.1:7979` for agent-driven testing.

## Logs

`~/Library/Logs/Tiramisu/Tiramisu.log` — tailable. Mirrored to in-app Debug Console (Debug menu).

## Generative fill backends

Two backends only:

| Backend | Service | Notes |
|---|---|---|
| `replicate` | `ReplicateFillService` | Cloud, requires API key (UserDefaults: `world.hanley.tiramisu.replicate.apiKey`) |
| `localFlux` | `LocalFluxFillService` | Spawns user-installed `mflux-generate-fill` subprocess (the reason sandbox is off) |

Switching is in-app: AI → Generative Fill Settings.
